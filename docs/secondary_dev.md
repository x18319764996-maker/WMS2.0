# 二次开发指南

## 推荐扩展顺序

1. 先补充环境配置和真实接口模板
2. 再补全页面对象 locator
3. 再补 Flow 层的业务编排
4. 最后补 AI 提示词和自愈策略细节

## 不要这样做

- 不要把 locator 直接散落在测试用例里
- 不要把接口请求写进页面对象里
- 不要把 AI 请求硬编码到每个页面方法中
- 不要跳过 Flow 层直接在 case 中拼步骤

## 推荐这样做

- 页面对象只关注页面能力
- 复杂控件统一沉淀到 `components/`
- 跨页面业务逻辑沉淀到 `flows/`
- 业务域规则沉淀到 `domains/`

## 注解规范

所有新增和修改的 Python 文件必须遵循以下注解标准：

### 必须添加的注解

1. **模块级 docstring**（文件顶部）— 用一句话说明模块职责，不使用 `"""模块说明。"""` 等泛泛描述
2. **类级 docstring** — 说明类的核心职责和关键能力
3. **方法/函数 docstring** — 一句话说明该方法的行为和作用
4. **`__init__` docstring** — 说明注入了哪些依赖
5. **行内注释** — 使用 `# 中文说明：` 前缀，解释非显而易见的逻辑

### 反模式清单

以下写法属于低质量注解，请避免：

| 反模式 | 示例 | 应替换为 |
|--------|------|----------|
| 泛化模块说明 | `"""模块说明。"""` | `"""WMS 登录页面对象，封装用户名/密码填写和登录提交操作。"""` |
| 泛化方法说明 | `"""执行操作。"""` | `"""填写用户名、密码并点击登录按钮。"""` |
| 泛化 init 说明 | `"""初始化。"""` | `"""注入 Playwright Page 和自愈定位策略。"""` |
| 缺少类 docstring | （类定义后无 docstring） | 添加说明类职责的中文 docstring |
