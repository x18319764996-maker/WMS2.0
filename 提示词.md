## 9. AI 能力要求

这是本项目的核心特色。请重点实现 AI 增强层，但要避免过度实验性设计。

### 9.1 Locator Strategy Layer

必须采用固定的分层定位策略：

1. 优先使用稳定规则定位
2. 规则定位失败时，尝试备用定位策略
3. 仍失败时，触发 AI 语义定位与自愈逻辑
4. 成功恢复后记录恢复依据、恢复结果和建议修复方案

必须支持以下定位思路：

- 显式 locator
- 文本语义
- label / placeholder / role / test id
- DOM 结构相似性
- 页面语义上下文
- 历史成功定位结果的候选恢复

### 9.2 Assertion Assistant

AI 不仅用于定位，还要用于断言辅助。必须支持：

- 辅助生成更稳的断言建议
- 识别页面语义异常信息
- 辅助验证复杂页面展示逻辑
- 结合流程上下文输出断言说明

但断言最终仍需以确定性逻辑为主，不允许只依赖模糊自然语言判断。

### 9.3 Failure Analysis Agent

失败后必须输出：

- 失败步骤
- 页面上下文摘要
- 关键控件状态
- 可能失败原因
- 修复建议
- 是否疑似定位失效
- 是否疑似环境异常
- 是否疑似数据问题

### 9.4 Report Enricher

必须把 AI 增强信息写入测试结果，例如：

- 自愈是否触发
- 触发位置
- 恢复成功还是失败
- 失败原因分析
- 修复建议
- 页面语义摘要

### 9.5 AI 治理要求

必须满足以下治理要求：

- AI 默认深度参与执行
- 允许 AI 在运行时做半自主决策
- 所有 AI 决策必须可解释
- 所有 AI 决策必须可关闭
- 所有 AI 决策必须保留日志
- 所有 AI 决策必须可审计

AI 日志中至少要记录：

- 输入上下文摘要
- AI 请求目的
- AI 返回结果摘要
- 最终采用的决策
- 是否成功
- 失败原因
- 回退动作

## 19. 明确禁止事项

- 只有 Page Object、没有 Flow 层
- 只有普通 Playwright 封装、没有 AI 分层
- 只有 UI 操作、没有 API 校验
- 只有测试代码、没有 README 和规范文档
- 只有概念说明、没有项目代码骨架
- 只在回复中贴代码，不落文件
- 只给抽象接口，不给默认实现骨架

## 20. 验收标准

- 能按业务流程维度输出报告
- 能让测试团队成员按模板快速新增用例
- 能作为长期维护框架继续扩展，而不是一次性脚本
- 生成完成后能执行最小验证并汇报结果
